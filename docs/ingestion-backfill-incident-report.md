# 文章采集卡住与历史回灌技术报告

## 结论

2026-05-28 首页机会数突然上涨，不是前端刷新生成了数据，而是云端后台任务真实向 `posts` 表插入了大量历史文章。数据库显示，当天新增 5165 条，其中 2026-05-28 14:00-14:53 CST 对应的 UTC 06 点这一小时插入 4996 条。

直接链路是：`enqueue_refresh_jobs` 为公众号源入队，`refresh_worker` 刷新 WeRSS 源后再入队 `sync_source_posts`，同步阶段按源抓取最近 `BACKEND_POST_FETCH_LIMIT=500` 条，并用 `upstream_post_id` 去重。此前未完整同步过的源，一旦队列恢复/提速，就会把最近窗口内的历史文章一次性补进本库。

## 证据

- 云端统计：`posts_total=8882`，默认可见 `8056`；2026-05-28 创建 `5165` 条，其中 `4687` 条进入默认可见集合。
- 插入峰值：2026-05-28 06:00-06:53 UTC，即 14:00-14:53 CST，`134` 个 sync job 抓取 `9110` 条，插入 `4996` 条，更新 `3292` 条，初筛丢弃 `822` 条。
- 主要来源包括：深圳招聘工作人才网 297、深圳招聘网 269、平安银行招聘 180、大学生英语竞赛平台 168、美团招聘 125。
- 这些来源的插入文章覆盖 2025、2024 甚至更早发布时间，说明是历史窗口回补，不是当天新发文章。
- 日志显示部署提速前 `refresh_worker` 长期以 `claimed=2` 为主；提速后出现 `claimed=10`，随后快速消化积压并触发历史回灌。
- `sync_jobs.trigger_type` 为 `schedule`，说明页面刷新只是看到了后台写入结果，不是浏览器触发插入。

## 根因

原有同步不是严格增量，而是“刷新源 -> 拉每源最近 N 条 -> 本库没见过就插入”。当队列处理能力不足时，系统一直在追旧任务；当吞吐突然提高时，之前没覆盖过的源会批量写入历史文章。

同时，正文抓取和 LLM 队列会反复扫描缺正文或未处理的历史内容，历史无关键时间文章也会进入首页默认计数，使“机会列表”数量显著膨胀。

## 已实施保护

- 为 `sources` 增加 `last_seen_published_at` 和 `last_seen_upstream_post_id` 高水位字段。
- 默认刷新改为增量模式：每源使用 `BACKEND_INCREMENTAL_POST_FETCH_LIMIT` 小窗口，并在遇到已见文章 ID 时停止；不再用固定天数截断低频公众号。
- 历史补库改为显式 `/api/jobs/backfill`，默认刷新不再自动做大窗口回补。
- `fetch_content` 和 `llm_post` 队列只处理近期发布或存在未来关键时间的内容，历史无时间内容不再挤占队列。
- 首页默认口径改为：未来关键时间内容 + 最近发布的无关键时间内容；历史内容仍可通过 `show_all=true` 查看。
- 新增 `/api/jobs/ingestion-health`，用于查看最近 24 小时新增曲线、来源增量、队列 backlog、最老活跃任务和最后成功同步。

## 后续观察

部署后至少观察 24 小时。正常状态下，每小时新增应保持平滑；无显式 backfill 时，不应再出现一次性几千条历史文章回灌。

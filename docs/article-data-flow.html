文章数据处理与分发流说明

这份页面解释当前系统如何从微信公众号聚合源拿文章，如何入库、初筛、补正文、交给 LLM 提取关键时间， 以及最终为什么某些文章会出现在首页“机会列表”里。它面向新加入项目的人，也面向需要排查线上数据波动的人。

默认同步：增量模式 历史补库：显式 backfill 首页：默认可见口径 监控：/api/jobs/ingestion-health
1. 总览 2. 采集与增量同步 3. 入库、初筛与投影 4. 正文与 LLM 队列 5. 首页分发与排序 6. 运维排查视角

1. 总览

系统不是直接把上游文章原样展示到首页。它会先把公众号源、原始文章、清洗后的文章、分类与时间信息分开存储。 首页只读后端 API 的分页结果，旁边的总数来自后端计算，不是前端本地统计。

1

采集层

负责刷新公众号源、抓文章列表、抓缺失正文。它关心“有没有新文章”和“正文是否完整”。

2

理解层

规则和 LLM 共同提取文章类型、分类、活动开始时间、截止时间、参与状态等结构化信息。

3

分发层

首页不展示全库，而是按默认可见口径、分类筛选、关键时间排序后分页返回。

2. 采集与增量同步

当前系统有两种同步模式：默认的 incremental 和显式的 backfill。 默认同步只看一个小窗口，用高水位判断“从哪里开始已经见过”；历史补库必须通过 POST /api/jobs/backfill 显式触发。

高水位字段是什么

高水位就是每个公众号源的同步书签。系统在 sources 表记录 last_seen_upstream_post_id 和 last_seen_published_at。 下一次同步时，如果文章列表里再次遇到这个已见文章，就知道后面大概率都是旧文章，可以停止继续处理。

字段 / 配置 当前作用 默认值或来源
last_seen_upstream_post_id 源级别文章书签，遇到它就停止继续向旧文章扫描。 sources 表
last_seen_published_at 源级别最新发布时间记录，用于理解同步进度和辅助排查。 sources 表
BACKEND_INCREMENTAL_POST_FETCH_LIMIT 默认增量同步每个源最多抓多少条文章。 100
BACKEND_POST_FETCH_LIMIT 显式 backfill 的默认大窗口，不用于普通增量刷新。 500
为什么之前会突然回灌历史文章： 旧逻辑接近“每个源拉最近 500 条，没见过就插入”。队列恢复或提速后，很多此前没完整覆盖的源被一次性同步，于是历史窗口里的文章一起入库。 现在默认路径已经变成小窗口增量，历史补库被隔离到显式 backfill。

3. 入库、初筛与投影

文章列表进入后，不会立刻变成首页卡片。系统先做规则初筛，保存原始载荷，再抽取正文、分类、时间和展示层级。 真正支撑首页排序的主要是 post_projections 表。

4. 正文与 LLM 队列

同步文章列表时，有些文章没有正文，或者需要 LLM 进一步提取时间与摘要。这两件事不直接阻塞列表同步， 而是通过 job_queue 异步处理。

队列资格

现在正文抓取和 LLM 不再无差别处理全库历史文章。只有满足下面任一条件才会继续排队或执行：

  • 发布时间在最近 BACKEND_QUEUE_RECENT_DAYS 天内，默认 30 天。这个字段只影响补正文和 LLM 队列，不影响公众号同步是否继续向下扫描。
  • 存在未来 deadline_at。
  • 存在未来 event_start_at。
这意味着历史无关键时间文章仍可留在数据库中，但不会继续抢正文抓取和 LLM 资源。低频社团公众号的采集停止点仍然是上一篇已见文章 ID，而不是这个天数阈值。
Job Type 谁创建 谁执行 作用
refresh_source enqueue_refresh_jobs 或 /api/jobs/refresh refresh_worker 刷新上游公众号源，随后创建增量同步任务。
sync_source_posts refresh_worker 或 /api/jobs/backfill refresh_worker 抓文章列表并入库。默认 incremental，显式 backfill 才大窗口。
fetch_content refresh_worker content_worker 为缺正文的文章补正文。
llm_post refresh_worker / content_worker llm_worker 调用 LLM 提取摘要、分类、活动开始、截止时间。

5. 首页分发与排序

首页调用 GET /api/posts，默认分页：前端第一次取 offset=0&limit=20， 加载更多再继续取下一页。机会列表旁边的数字是后端返回的 total，不是前端本地计算。

关键时间排序规则

sort=deadline 这个名字保留了旧 API，但现在排序看的不是单一截止时间，而是“关键时间”。 关键时间只从 deadline_at 和 event_start_at 中选择，二者同权重。

  1. 第一组：有未来关键时间的文章，按最近的未来关键时间从近到远。
  2. 第二组：没有截止时间、没有活动开始时间的文章，按发布时间从早到晚。
  3. 第三组：关键时间都过期的文章，排最后，按最近过期优先。

默认首页为什么少于全库

当前设计刻意把“首页机会列表”与“数据库全量文章”分开。全库可能保留历史文章、已过期文章、无时间文章； 首页默认只展示更像机会流的内容。需要排查全库时，用 show_all=true。

6. 运维排查视角

如果以后又出现“首页数量突然变化”“两天没新文章”“队列积压”等现象，优先看下面几个入口。

GET /api/posts?limit=1
全库口径
GET /api/posts?limit=1&show_all=true

GET /api/jobs/ingestion-health
队列摘要
GET /api/jobs/summary
显式补库
POST /api/jobs/backfill
现象 优先检查 如何理解
首页总数突然上涨 /api/jobs/ingestion-health 的 posts_inserted_by_hour_24h 如果某小时插入很多，说明是后台入库导致；再看 top_sources_inserted_24h 确认来源。
好像没有新文章 last_successful_sync、oldest_active_job、queue_by_type_status 判断是同步没跑、refresh 积压、sync 积压，还是上游本身没有新内容。
首页比全库少很多 对比 /api/posts 与 /api/posts?show_all=true 这是默认分发口径在生效：历史、过期、老的无时间内容不会全进首页。
LLM 处理很慢 llm_post pending 数、LLM worker 日志 LLM 是限速慢任务；现在只处理近期或有未来关键时间的候选，避免全库重扫。
这套结构的核心原则是：默认路径只稳定抓新文章；历史文章要显式补库；首页展示机会流，不等于全库文章列表。
